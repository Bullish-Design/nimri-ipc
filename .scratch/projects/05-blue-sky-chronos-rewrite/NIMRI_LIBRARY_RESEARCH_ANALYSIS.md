## What the upstream `niri_ipc` protocol demands from `nimri-ipc`

The public ŅłĆurlŅłéniri_ipc crate docsŅłéhttps://docs.rs/niri-ipc/latest/niri_ipc/Ņłü show that Niri IPC is a newline-delimited JSON protocol over a Unix socket whose path comes from `$NIRI_SOCKET`. Clients send one `Request` per line and receive one `Reply` per line. After `Request::EventStream`, the compositor stops reading additional requests on that socket and continuously emits `Event`s instead; the docs explicitly say you need **two IPC sockets** if you want to read the event stream and send more requests concurrently. That single detail should drive your core session design: `nimri-ipc` needs a **control socket** and an **event socket** as first-class concepts, not as an afterthought. ŅłĆciteŅłéturn12view0Ņłéturn13view0Ņłü

The same docs also warn that requests are processed separately and time passes between them, so adjacent requests like `Workspaces` and `Windows` may not describe a consistent moment in time, and chained actions like focus-then-close can race with compositor state changes. That means the right mental model is **event-sourced cached state**, not ŌĆ£query twice and hope the snapshots line up.ŌĆØ `nimri-ipc` should therefore expose two surfaces: a low-level request/reply client for rare or imperative operations, and a reducer-backed state store fed from the event stream for everything user-facing. `nirip` should be built overwhelmingly on the second surface. ŅłĆciteŅłéturn12view0Ņłéturn13view0Ņłéturn14view3Ņłü

The upstream surface area is also large enough that hand-maintaining it will become a drift tax. The docs list a broad `Request` set including version, outputs, workspaces, windows, layers, keyboard layouts, focused output/window, picking operations, output reconfiguration, overview state, screencasts, and arbitrary compositor `Action`s. `Response` and `Event` are similarly broad, and `Action` itself is extensive enough to fill several screens of focus, move, monitor, workspace, screenshot, output, and overview operations. That argues strongly for a generated or semi-generated protocol layer in Nim rather than a purely manual translation. ŅłĆciteŅłéturn13view0Ņłéturn14view0Ņłéturn14view1Ņłéturn14view2Ņłü

One especially useful upstream clue is that the Rust crate offers an optional `json-schema` feature that derives JSON Schema for its types, and the crate documentation warns that it follows the Niri version rather than Rust semver stability, with new fields and enum variants possible even in patch bumps. The clean rewrite path is therefore to make `nimri-ipc` **version-aware** and **schema-driven**: pin and test against exact upstream Niri protocol versions, generate Nim protocol artifacts from an upstream schema snapshot or equivalent exported metadata, decode leniently for additive fields, and preserve an escape hatch for unknown payloads so that additive upstream changes do not immediately break the stack. ## The cleanest blue sky architecture for the three libraries

The clean rewrite should make `nimri-ipc` a **protocol-and-session library**, not an app framework. Its job is to know how to connect, encode/decode, subscribe, reduce event streams into cached state, reconnect cleanly, and expose a precise typed API. It should not know about widgets, theming, layout, user preferences, or ŌĆ£business logicŌĆØ rules beyond what is necessary to represent NiriŌĆÖs protocol faithfully. Put differently: if `sidebard` disappeared tomorrow, `nimri-ipc` should still make sense as a standalone Niri IPC client library. That is consistent with the upstream protocolŌĆÖs own separation between transport, event stream, and helper state modules. ŅłĆciteŅłéturn12view0Ņłéturn14view3Ņłü

`nirip` should be the **Niri domain model and ergonomics layer**. It should consume `nimri-ipc`ŌĆÖs typed state/events and produce app-friendly selectors and operations: focused-window streams, workspace inventories, urgency views, monitor/workspace projections, semantic commands, derived labels, rate-limited updates, and higher-level policies that are still compositor-domain rather than presentation-domain. It should also be the layer where you hide compatibility quirks between Niri protocol versions, because the upstream crate is intentionally version-tied and not semver-stable. `sidebard` should talk to ŌĆ£focused window changedŌĆØ and ŌĆ£workspace summaryŌĆØ concepts, not directly to raw event enums wherever possible. ŅłĆciteŅłéturn12view0Ņłéturn14view1Ņłéturn14view3Ņłü

`sidebard` should be a **presentation shell**, not a protocol client. It should depend on `nirip`, hold view models, own rendering/layout/theme/config, and contain app-specific policy like what to show, when to animate, what to hide, and how to expose user commands. If it has multiple render targets in the future, such as a GTK build and some other shell-native renderer, those should be `sidebard` adapters around the same `nirip` domain layer. The cleanest codebase is the one where widget code never parses JSON, never reconnects sockets, and never hand-assembles compositor requests. ŅłĆciteŅłéturn23search0Ņłéturn25search0Ņłéturn29search5Ņłéturn29search7Ņłü

Across all three libraries, standardize on **one async story, one error story, one serialization story, and one logging story**. That means Chronos for async, explicit `Result`-style library boundaries for recoverable failures, `nim-json-serialization` for wire/persistence codecs, and Chronicles for logs. The ŌĆ£best, cleanest, most elegantŌĆØ outcome is mostly about removing conceptual branching: no mixed runtimes, no mixed logging systems, no mixed codecs, no mixed error philosophy. ### `nimri-ipc` rewrite plan

Implement `nimri-ipc` in internal layers. The first layer should be **raw protocol types**: `Request`, `Response`, `Event`, `Action`, plus all constituent objects such as outputs, windows, workspaces, layouts, overview, keyboard layouts, screencasts, and so on. Prefer code generation or schema-assisted generation here; do not hand-maintain a huge tagged-union surface unless you have a very strong reason. The second layer should be a **codec layer** built on `nim-json-serialization`. Give it two modes: strict encoding for outgoing requests and tolerant decoding for incoming payloads. Tolerant decoding should preserve unknown or forward-added material where practical so that additive upstream protocol changes degrade gracefully. Also support ŌĆ£raw JSON fallbackŌĆØ for unknown enum-like cases if your generator cannot fully future-proof tagged unions. That design is directly motivated by upstreamŌĆÖs versioning model. ŅłĆciteŅłéturn25search0Ņłéturn12view0Ņłü

The third layer should be a **Chronos transport layer** for Unix-domain sockets with newline framing. Model three concrete types: `ControlConnection`, `EventConnection`, and `Session`. `Session` should own reconnection policy, backoff, version handshake, initial event subscription, and state invalidation / revalidation semantics. On reconnect, it should reopen both sockets, re-request the event stream, and treat cached state as provisional until the full initial state events arrive again. Because event-stream startup gives a handled reply and then full-state events up front, this is the cleanest rebuild model. ŅłĆciteŅłéturn13view0Ņłéturn14view3Ņłéturn22search3Ņłü

The fourth layer should be a **reducer and cache layer**. Expose immutable snapshots plus targeted subscriptions, not mutable shared objects. ŅłĆciteŅłéturn13view0Ņłéturn14view3Ņłü

The fifth layer should be the **public client API**. I would expose four surfaces: a raw command client, a raw event stream, a cached state store, and a thin convenience facade. Testing should include golden-wire tests for request/response/event JSON, reducer correctness tests driven by recorded event traces, reconnect tests with simulated socket drops, and compatibility tests against frozen upstream schema snapshots. ### `nirip` rewrite plan

Make `nirip` the domain adapter that turns raw compositor state into **stable semantic APIs**. Its primary abstraction should be an owning service object, something like `NiriDomainRef`, backed by a `nimri-ipc` session and its cached state. The job of this service is to answer questions that the UI actually asks: what is focused, what is visible, what changed, what is urgent, what labels should be shown, which workspace is active on which output, and what command should be emitted for a particular semantic intent. ### `sidebard` rewrite plan

Treat `sidebard` as an application shell with a unidirectional data flow: `nimri-ipc` session ŌåÆ `nirip` domain service ŌåÆ `sidebard` view models ŌåÆ renderer/widgets. If `sidebard` is GTK/libadwaita-based, choose either `gintro` or Owlkettle early and commit to one mental model. `gintro` is the more direct binding-oriented path; Owlkettle is the more declarative path. ## Open questions and limitations

I was able to ground the rewrite recommendations in the public upstream `niri_ipc` specification and in official or primary documentation for Chronos, Chronicles, the Status/Nimbus library ecosystem, `cligen`, GTK-related Nim frameworks, and KDL. I was **not** able in this environment to retrieve the current source trees for the Bullish-Design repositories themselves, nor to retrieve the later uploaded design/planning documents after the initial attached Chronos rewrite concept. That means the architecture above is a high-confidence blue-sky plan, but not a line-by-line audit of the current Bullish-Design code.

The biggest unresolved technical question is `sidebard`ŌĆÖs rendering/shell target: if it is fundamentally a GTK/libadwaita application, the framework choice is straightforward; if it needs deeper Wayland shell semantics, the binding story needs one more focused validation pass before implementation begins.

The biggest product question is how much of Niri-specific policy belongs in `nirip` versus `sidebard`. My recommendation is conservative: keep `nirip` compositor-domain and keep presentation policy in `sidebard`. If your uploaded design docs argue for a heavier `nirip`, I would still only move logic downward when it clearly generalizes beyond one UI.
